# Schema regression tests (Schemathesis)

Property-based fuzzing of the OpenAPI spec, run on every PR against
`atomicmemory-core`. Catches the "wire shape drifted but route tests
still pass" class of regression — for example, a route that drops a
declared response field, or starts returning a different status code
for a known input shape.

## How it works

1. `bash run-schemathesis.sh` brings up `docker-compose.smoke-isolated.yml`
   (stub-LLM mode, isolated postgres named volume, port 3061).
2. The script polls `GET /health` until the app is up.
3. `schemathesis run` reads `../../openapi.yaml`, generates fuzz inputs
   for every declared path/operation, and runs three checks against
   each response:
   - **`status_code_conformance`** — response's status code is one
     declared in the spec for that operation.
   - **`response_schema_conformance`** — response body matches the
     declared schema for that status code.
   - **`content_type_conformance`** — response's `Content-Type` is
     one of the declared content types.
4. Compose stack is torn down (with `--volumes` so re-runs start
   from a clean DB).

The default `negative_data_acceptance` check is **suppressed** — fuzz
inputs that produce 4xx are correct behavior for atomicmemory-core, not
a bug. We only care about shape-level regressions.

## Running locally

```bash
# From the repo root:
pip install schemathesis==3.*
npm run test:schema
# Or directly:
bash tests/schema/run-schemathesis.sh
```

The driver brings the smoke-isolated stack up and tears it down on
exit. If a previous run left containers around, the up/down cycle
cleans them up.

If port 3061 is occupied locally, set `APP_PORT=<free-port>` before
running and the smoke compose's `${APP_PORT}` substitution will pick
it up.

## Real failure vs flake

A **real** `response_schema_conformance` failure looks like:

```
1. Test Case ID: ...
   Check           : response_schema_conformance
   Body            : {"memories": [...]}
   Path            : /v1/memories/list
   Status Code     : 200

   <error: 'episode_id' is a required property>
```

That means a route's response is missing a field declared as required
in `openapi.yaml`. Fix is either to add the field back to the response
or remove `required: [episode_id]` from the spec (then re-run
`npm run check:openapi` to regenerate).

A **flake** typically looks like:
- A timeout (>5s/example) on an LLM-dependent endpoint
  (`POST /v1/memories/ingest`, full `/v1/memories/search`). The
  stub LLM key returns errors quickly, but if the embedding warmup
  hasn't finished, the very first call can be slow. Re-run; if it
  re-flakes, narrow the run with `--include-path` filters to skip
  warmup-sensitive endpoints.
- A 500 on an endpoint that's not in the spec at all. Schemathesis
  will flag the 500 as `status_code_conformance` because no 5xx
  response is declared. The fix there is to declare the realistic
  error envelope in `openapi.yaml`, not to ignore the failure.

## CI artifact on failure

CI uploads `schemathesis-report.json` (Schemathesis.io tarball),
`schemathesis-report.xml` (JUnit), and `schemathesis-cassette.yaml`
(VCR cassette for failure replay) from the repo root as a single
`schemathesis-diagnostics` artifact when the job fails. Download
it from the run page to get the full per-test-case breakdown
without re-running locally.

## Updating example coverage

`HYPOTHESIS_MAX_EXAMPLES=50` in `run-schemathesis.sh` keeps the
bake-window CI runtime under ~3-5 minutes. Once the baseline is
stable, raise it (100, 200) for deeper coverage. The script is the
authoritative source for these tuning values — Schemathesis 3.x's
CLI does not consume a separate ini config file, so we don't keep
one (it would be a drift trap).

## Status

- **`continue-on-error: true` for the bake window.** First 1-2 weeks
  in CI, this job surfaces signal but does not block merge. Promotion
  to a hard gate is a separate, follow-up PR.
- The OpenAPI spec is auto-generated by `npm run check:openapi`
  (CI-gated). Schemathesis fuzzes whatever the generator produced —
  it will not catch "schema is wrong" issues, only "implementation
  drifted from schema" issues.
